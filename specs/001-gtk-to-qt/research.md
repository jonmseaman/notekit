# Research: GTK3 to Qt Migration

**Branch**: `001-gtk-to-qt` | **Date**: 2026-03-02

## §1 Qt Text Editor for Markdown Editing

**Question**: Which Qt text editing component best replaces `Gsv::View` (GtkSourceView) for NoteKit's markdown editor with proximity rendering?

**Decision**: `QTextEdit` (not `QPlainTextEdit`)

**Rationale**:
- `QTextEdit` supports embedded `QWidget` objects via `QTextEdit::setDocument` and `createStandardContextMenu`, and uses `QTextDocument` as its backing model — required for embedded checkboxes and LaTeX widgets.
- `QPlainTextEdit` is optimised for plain text and does not support embedded objects; it cannot replicate NoteKit's inline widget anchors.
- `QTextEdit` + custom `QSyntaxHighlighter` subclass is the standard Qt pattern for syntax-highlighted editors.
- GtkSourceView's language definition files (`.lang`) are not portable; a custom `MarkdownHighlighter : QSyntaxHighlighter` implementing NoteKit's existing highlight rules is the replacement.

**Alternatives Considered**:
- `QScintilla`: Third-party, adds an external dependency — violates Principle V.
- `KTextEditor` (KDE): Heavy KDE dependency — violates Principle V.
- `QPlainTextEdit`: Cannot embed widgets — eliminates checkbox and LaTeX functionality.

## §2 Embedded Widgets in Text (Checkboxes, LaTeX, Images)

**Question**: How to replicate GTK's child-anchor embedded widgets (`gtk_text_buffer_create_child_anchor`) in Qt?

**Decision**: `QTextCharFormat` + custom `QTextObjectInterface` for rendering; `QTextEdit::createMimeDataFromSelection` for clipboard.

**Rationale**:
- Qt's `QTextDocument` supports inline objects via `QTextCharFormat::setObjectType` + `QTextObjectInterface::drawObject`. This is the documented Qt mechanism for custom inline rendering.
- For checkboxes: implement a `CheckboxTextObject : QObject, QTextObjectInterface` that renders a checkbox glyph and responds to mouse clicks via `QTextEdit::mousePressEvent`.
- For LaTeX/image widgets: implement `ImageTextObject : QObject, QTextObjectInterface` that renders a `QPixmap` generated by cLaTeXMath at the correct text baseline.
- The `QTextObjectInterface::intrinsicSize` method provides accurate layout metrics, ensuring text flows correctly around embedded objects.

**Alternatives Considered**:
- `QTextEdit::setWidget` (widget anchors): Qt does not have a direct child-anchor equivalent; `QTextObjectInterface` is the correct replacement.
- Absolutely positioned `QWidget` overlays: Fragile with scroll position, does not integrate with text layout engine.

## §3 Proximity Rendering (Hide Markdown Markup Near Cursor)

**Question**: How to replicate GtkSourceView's proximity rendering (markdown symbols hidden at a distance from the cursor, shown near it)?

**Decision**: `QSyntaxHighlighter` with cursor-position-aware formatting via `cursorPositionChanged` signal.

**Rationale**:
- `QTextEdit` emits `cursorPositionChanged` on every cursor move.
- `QSyntaxHighlighter::rehighlightBlock` can be called on the block containing the cursor and its neighbours, updating `QTextCharFormat` to show/hide markup tokens.
- The `QTextCharFormat::setForeground(Qt::transparent)` or `QTextCharFormat::setFontFixedPitch(false)` + zero-width trick replicates the "invisible markup" tag used by the GTK version.
- Performance: only affected blocks are re-highlighted on cursor move, not the full document.

**Alternatives Considered**:
- Full document rehighlight on every cursor move: O(n) per keystroke — too slow for large notes.
- HTML-based rendering (QTextBrowser): Read-only; cannot edit.

## §4 Tablet / Stylus Pressure Input

**Question**: How to replicate `GdkDevice` pressure-sensitive input (`gdk_event_get_axis(event, GDK_AXIS_PRESSURE)`) in Qt?

**Decision**: `QTabletEvent` (received in `QWidget::tabletEvent`)

**Rationale**:
- Qt 6 delivers pressure, tilt, rotation, and tangential pressure via `QTabletEvent`. These map directly to NoteKit's `CStroke` pressure values.
- `QTabletEvent::pressure()` returns a `qreal` in [0.0, 1.0] — the same range as GTK's `GDK_AXIS_PRESSURE`.
- Mouse fallback: When no tablet is present, `QMouseEvent` is used with a fixed pressure of 1.0, identical to current GTK behaviour.
- `QWidget::setAttribute(Qt::WA_TabletTracking)` enables tablet events even without button press (hover pressure).

**Alternatives Considered**:
- Platform-native tablet APIs (WinTab on Windows, IOKit on macOS): Violate Principle I (Qt-First UI) and Principle II (platform-specific ifdefs).
- Using only `QMouseEvent`: Loses pressure sensitivity — violates FR-004.

## §5 zlib vs QByteArray::qCompress

**Question**: Can `QByteArray::qCompress` replace the direct `zlib` dependency?

**Decision**: **Retain `zlib` directly.** Do not replace with `QByteArray::qCompress`.

**Rationale**:
- `QByteArray::qCompress` prepends a 4-byte big-endian uncompressed-size header before the zlib stream. This is a Qt-specific extension and produces data that is **not** compatible with raw zlib streams.
- NoteKit's drawing data is stored in the existing note files using raw zlib compression. Switching to `QByteArray::qCompress` would produce files incompatible with the GTK3 version — violating FR-013 (byte-for-byte format compatibility).
- Keeping `zlib` directly preserves the existing serialisation path unchanged.

**Alternatives Considered**:
- `QByteArray::qCompress` with custom header stripping: Fragile and non-obvious; not worth the complexity to eliminate one small dependency.
- Remove compression entirely: Would change file format — not acceptable.

## §6 Thread-Safe GUI Signaling (Replacing Glib::Dispatcher)

**Question**: How to replace `Glib::Dispatcher` (thread→main-thread callback) for find-in-files results?

**Decision**: `QMetaObject::invokeMethod(..., Qt::QueuedConnection)` or direct Qt signal/slot with `Qt::QueuedConnection`

**Rationale**:
- Qt's signal/slot mechanism automatically marshals cross-thread signal emissions via the event queue when the sender and receiver are in different threads. Connecting a signal from a `QThread`-based worker to a slot on the main-thread widget with `Qt::QueuedConnection` (or `Qt::AutoConnection` which selects `Queued` automatically for cross-thread) is the exact equivalent of `Glib::Dispatcher`.
- `CFindInFiles` refactors to `QObject` with a `QThread` worker emitting a `resultFound(QString path, QString context)` signal. The main window connects to this signal with `Qt::QueuedConnection` — no manual synchronisation needed.

**Alternatives Considered**:
- `QMutex` + polling timer: More complex than needed; Qt's signal/slot mechanism eliminates the need.
- `std::thread` + `QMetaObject::invokeMethod`: Valid, but converting to `QThread` + signals is cleaner and more idiomatic.

## §7 CMakeLists.txt Structure for Qt 6.10

**Question**: What is the correct CMake setup for Qt 6.10 with all required modules?

**Decision**: `find_package(Qt6 REQUIRED COMPONENTS Core Gui Widgets Concurrent)` with `qt_add_executable` and `qt_add_resources`.

**Rationale**:
- Qt 6.10 requires CMake ≥ 4.2 (per constitution). `qt_add_executable` is the modern replacement for `add_executable` + manual Qt boilerplate — it handles MOC, UIC, and RCC automatically.
- `qt_add_resources` compiles `.qrc` files into the binary at build time, eliminating the runtime `data/` directory path lookup.
- `Qt6::Concurrent` provides `QtConcurrent::run` as an alternative threading API for find-in-files if preferred over `QThread`.

**Modules required**:
```cmake
find_package(Qt6 REQUIRED COMPONENTS Core Gui Widgets Concurrent)
find_package(PkgConfig REQUIRED)
pkg_check_modules(JSONCPP REQUIRED jsoncpp)
pkg_check_modules(ZLIB REQUIRED zlib)
if(UNIX AND NOT APPLE)
    pkg_check_modules(FONTCONFIG REQUIRED fontconfig)
endif()
# cLaTeXMath: vendored/submodule — add_subdirectory or FetchContent
```

## §8 Settings Migration Strategy

**Question**: How to migrate GSettings/JSON backend to Qt without breaking existing user configurations?

**Decision**: `QSettings` with INI format on all platforms; migrate existing JSON config values on first launch.

**Rationale**:
- `QSettings` provides a cross-platform abstraction (registry on Windows, plist on macOS, INI on Linux) without dconf/GSettings dependencies.
- The existing `default_config.json` and current JSON settings file are read once at first launch of the Qt build, converted to `QSettings` format, and the JSON file is left untouched (for GTK3 backwards compatibility during the transition).
- Settings keys map 1:1: `base-path`, `active-document`, `colors`, `csd`, `sidebar`, `presentation-mode`, `syntax-highlighting`.

**Alternatives Considered**:
- Continue reading JSON directly: Viable but bypasses `QSettings`'s per-platform storage conventions and sync semantics.
- Use a different config library: Unnecessary — `QSettings` is sufficient and already in Qt Core.

## §9 Asset Migration (Icons, Stylesheet, Fonts)

**Question**: How to migrate GTK runtime asset loading to Qt?

**Decision**: Bundle all assets into a Qt resource file (`.qrc`) compiled into the binary.

**Rationale**:
- The GTK version searches runtime paths (`/usr/share/notekit/data`, CWD) to locate icons, stylesheets, and fonts. This is fragile and requires a correct installation.
- Qt's resource system (`:/path/to/asset`) provides zero-configuration asset loading regardless of installation layout.
- `stylesheet.css` is converted to `stylesheet.qss` — Qt's stylesheet syntax is similar but uses Qt widget selectors instead of GTK widget names.
- Custom fonts loaded via `QFontDatabase::addApplicationFont(":/fonts/...")`.
- Application icon: `QApplication::setWindowIcon(QIcon(":/icons/..."))`.

**Files to migrate**:
- `data/preferences.glade` → Removed; preferences UI built in C++ or via Qt Designer `.ui`
- `data/toolbar.glade` → Removed; toolbar built in C++ via `QToolBar::addAction`
- `data/stylesheet.css` → `data/stylesheet.qss` (translated rule-by-rule)
- `data/icons/` → Compiled into `resources.qrc`
- `data/fonts/` → Compiled into `resources.qrc`
- `data/default_config.json` → Compiled into `resources.qrc`
- `data/gschemas.compiled` → Deleted (GSettings schema, not needed for Qt)
